Allgemein

Releases des MyCoRe -Projektes erscheinen zweimal pro Jahr: ein Juni-Release und ein Dezember-Release. Der garantierte Supportzeitraum für ein Release beträgt ein Jahr nach dem Erscheinen. In Ausnahmefällen können auch nach diesem Zeitraum noch kritische Fixes eingespielt werden. Die Versionsnummern entsprechen den Tags in GitHub und beginnen mit v. Branches haben die Form yyyy.06.x bzw. yyyy.12.x.

Release-Zyklus

Pro Jahr werden zwei Releases erstellt. Die Release-Zeitpunkte werden mit den Entwicklertreffen synchronisiert:

  • Juni-Release, Entwicklertreffen kurz vor oder nach dem Release
  • Dezember-Release, Entwicklertreffen kurz vor oder nach dem Release

Sechs Wochen vor einem Release wird der Release-Branch vom Main abgezweigt. Ab diesem Zeitpunkt werden in den LTS-Branches keine umfangreichen Änderungen mehr zugelassen.

Phasen zwischen zwei Releases

Zeitpunkt Aktivität
Ab Release der Vorversion Beginn des neuen Entwicklungszyklus im Main
Bei nächster Telko Durchsicht der Draft-PRs
bis 8 Wochen nach Alt-Release
  • PRs mit hohem Risiko erlaubt
  • Major-Updates bei JS- und Java-Dependencies
bis 8 Wochen vor Release Dependabot-JS-PRs mit Minor-Updates
6 Wochen vor Release
  • Release-Branch abzweigen
  • keine PRs mit mittlerem oder hohem Risiko mehr in LTS-Branches
Tag des Releases Release wird erstellt und in das zentrale Maven-Repository hochgeladen

Kategorisierung von Pull Requests

Pull Requests werden nach Umfang und Risiko in drei Kategorien eingeteilt. Diese Einteilung bestimmt, in welcher Phase des Release-Zyklus ein PR in den Main aufgenommen werden darf.

gering mittel hoch
Bugfix Update von Major-Versionen der Dependencies PMD-Änderungen
Update von Minor-Versionen der Dependencies neue Features Architekturänderungen
nicht-umfangreiche und nicht-tiefgehende API-Änderungen umfangreiche oder tiefgehende API-Änderungen
einfache Migration erforderlich umfangreiche Migration erforderlich

gering – risikoarm, jederzeit einspielbar

  • Bugfixes ohne API-Änderungen
  • Patch- und unkritische Minor-Updates von Dependencies
  • Interne Refactorings
  • Performance-Optimierungen ohne Verhaltensänderung
  • Dokumentationsänderungen
  • Logging- und Monitoring-Anpassungen

Eigenschaften:

  • Keine oder triviale Migration
  • Rückwärtskompatibel
  • Geringes Risiko
  • Rollback trivial

mittel – erhöhtes Risiko, mit Bedacht einspielen

  • Neue Features (aktiviert oder relevant für Nutzer)
  • Major-Version-Updates von Dependencies
  • Nicht-tiefgehende API-Änderungen
  • Minor-Updates mit Behavior-Changes
  • Konfigurationsänderungen
  • Security Fixes mit Seiteneffekten
  • Entfernen von Deprecated APIs (mit Ersatz)
  • Performance-Änderungen mit messbarem Impact

Eigenschaften:

  • Einfache Migration
  • Teilweise Breaking Potenzial
  • Tests und Doku empfohlen
  • Rollback möglich, aber nicht trivial

hoch – hohes Risiko, sorgfältige Abstimmung erforderlich

  • Breaking Changes
  • Umfangreiche oder tiefgehende API-Änderungen
  • Architekturänderungen
  • Datenmodell- oder Persistenzänderungen
  • Verhaltensänderungen bestehender Features
  • Entfernen rückwärtskompatibler Pfade
  • Große Security-Fixes (Auth, Crypto, Permissions)

Eigenschaften:

  • Umfangreiche Migration erforderlich
  • Nicht rückwärtskompatibel
  • Hoher Test- und Abstimmungsaufwand
  • Rollback schwierig oder unmöglich

Versionsnummern

Releases werden unter ihrer Nummer yyyy.06.fix bzw. yyyy.12.fix (z. B. 2026.06.1) auf https://central.sonatype.com/artifact/org.mycore/mycore abgelegt. Die ersten beiden Bestandteile der Versionsnummer entsprechen dem geplanten Erscheinungsmonat des Releases (Juni bzw. Dezember). Der dritte Bestandteil (fix) zählt Bugfix-Releases innerhalb eines LTS-Branches.

Main

Der Main ist der aktuelle Entwicklungszweig. Nach erfolgreicher Übernahme eines Developer-Branches in den Main wird nach dem CI-Test ein entsprechender Snapshot bereitgestellt.
